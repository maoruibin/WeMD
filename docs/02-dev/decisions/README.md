## 📝 决策记录

### ADROO-001: 采用 markdown-it 作为解析器

**状态**: ✅ 已接受

**上下文**: 需要 Markdown 解析器，有多种选择

**决策**: 采用 markdown-it

**理由**:
- 生态成熟，插件丰富
- 性能优秀
- 易于扩展

**替代方案**: marked、remark

---

### ADROO-002: 采用 Zustand 作为状态管理

**状态**: ✅ 已接受

**上下文**: 需要状态管理方案

**决策**: 采用 Zustand

**理由**:
- API 简洁
- 无需包裹 Provider
- TypeScript 友好
- 包体积小

**替代方案**: Redux、MobX、Jotai

---

### ADROO-003: 采用 Monorepo 架构

**状态**: ✅ 已接受

**上下文**: 多应用共享核心代码

**决策**: 采用 pnpm + Turborepo

**理由**:
- 代码复用
- 统一构建
- 依赖管理清晰

**替代方案**: Multi-repo、Nx

---

### ADROO-004: 本地优先的存储策略

**状态**: ✅ 已接受

**上下文**: 用户数据存储方案选择

**决策**: 默认使用 IndexedDB，可选 File System API

**理由**:
- 保护用户隐私
- 无需登录
- 离线可用

**替代方案**: 纯云端存储
